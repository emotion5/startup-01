# WorkHub 사용자 가이드

## 개요
WorkHub는 GitHub의 브랜치 개념을 회사 업무에 적용한 협업 도구입니다.
팀원들이 각자의 작업을 브랜치에서 진행하고, 완료된 작업을 상위 브랜치로 머지하여 회사 전체 진행상황에 반영합니다.

---

## 1. 기본 개념

### 브랜치 (Branch)
- **회사 메인 브랜치**: 전체 회사의 최상위 브랜치 (마스터)
- **부서/프로젝트 브랜치**: 개발팀, 마케팅팀 등의 중간 브랜치
- **팀 브랜치**: 특정 기능이나 캠페인을 위한 하위 브랜치
- **개인 작업 브랜치**: 개인이 담당하는 구체적 작업

### 커밋 (Commit)
- 완료된 작업 내역을 기록하는 것
- 제목과 설명으로 구성
- "공시된 업무 내역"으로 팀원 모두가 확인 가능

### 머지 (Merge)
- 하위 브랜치의 작업을 상위 브랜치로 통합
- 팀장/관리자의 승인 필요
- 승인되면 상위 브랜치에 반영되어 회사 대시보드에 표시

## 2. 업무 프로세스

### Phase 1: 프로젝트 연결 및 브랜치 확인

#### 2.1. WorkHub 진입
1. 메인 대시보드에서 프로젝트 카드 클릭
2. 3D 전시관에서 프로젝트 키오스크 클릭
3. 정보 모달에서 **"📋 프로젝트 내용보기"** 버튼 클릭
4. WorkHub 페이지로 이동

#### 2.2. 브랜치 구조 파악
```
WorkHub 메인 화면
├── 연결된 프로젝트 정보 (상단)
├── Branch Overview (화이트보드)
│   └── 브랜치 트리 시각화
│       - Main (회사 메인)
│       ├── Development (개발팀)
│       │   ├── Feature-A (기능 A)
│       │   └── Feature-B (기능 B)
│       └── Marketing (마케팅팀)
│           └── Campaign-1 (캠페인 1)
└── Pull Requests (머지 요청 목록)
```

**확인 사항:**
- 현재 프로젝트의 브랜치 구조
- 각 브랜치의 상태 (active/merged)
- 진행 중인 Pull Request

---

### Phase 2: 작업 수행 및 커밋

#### 2.3. 작업할 브랜치 선택
1. 화이트보드에서 자신이 작업할 브랜치 노드 클릭
2. 브랜치 상세 페이지로 이동

#### 2.4. 작업 내역 커밋 (공시)
**브랜치 상세 페이지 구성:**
```
┌─────────────────────────────────────────┐
│ [← Back to Whiteboard]                  │
├─────────────────────────────────────────┤
│ 브랜치 정보                               │
│ - 이름: Feature-A                        │
│ - 타입: team                             │
│ - 상태: active                           │
│ - [Merge to Parent] 버튼                 │
├─────────────────────────────────────────┤
│ 커밋 입력창                               │
│ ┌─────────────────────────────────────┐ │
│ │ 제목: [               ]              │ │
│ │ 설명: [                              │ │
│ │        (textarea)                    │ │
│ │       ]                              │ │
│ │ [커밋하기] [취소]                      │ │
│ └─────────────────────────────────────┘ │
├─────────────────────────────────────────┤
│ Commit History (공시 내역)               │
│ - 2025-01-15: 로그인 API 연동 완료        │
│ - 2025-01-14: 사용자 인증 로직 구현        │
│ - 2025-01-13: DB 스키마 설계              │
└─────────────────────────────────────────┘
```

**커밋 작성 단계:**
1. **제목 입력**: 완료한 작업을 한 줄로 요약
   - 예: "로그인 API 연동 완료"
2. **설명 입력**: 작업 내용을 상세히 기록
   - 예: "Firebase Authentication을 이용한 이메일/비밀번호 로그인 기능 구현. 에러 핸들링 및 토큰 관리 로직 추가."
3. **커밋하기 버튼 클릭**
4. 커밋이 브랜치에 추가되고 히스토리에 표시됨

**커밋 시점:**
- 하나의 의미있는 작업이 완료되었을 때
- 일일 업무 종료 시
- 주간 보고 전
- 팀장에게 검토를 요청하기 전

---

### Phase 3: 검토 요청 및 승인

#### 2.5. 머지 요청 생성 (Pull Request)
**상황:** Feature-A 브랜치의 작업이 완료되어 Development 브랜치로 합치고 싶을 때

1. 브랜치 상세 페이지에서 **"Merge to Parent"** 버튼 클릭
2. 머지 요청 생성
   - 자동으로 부모 브랜치(Development)를 타겟으로 설정
   - 현재 브랜치의 모든 커밋이 포함됨
3. 상태가 "pending"으로 표시
4. 팀장/관리자에게 알림 (향후 기능)

#### 2.6. 머지 요청 검토 (팀장/관리자)
**WorkHub 메인 화면의 Pull Requests 섹션:**
```
Pull Requests
├── [Pending] Feature-A → Development
│   작성자: 홍길동
│   커밋 3개
│   [승인] [거절] 버튼
│
├── [Pending] Campaign-1 → Marketing
│   작성자: 김영희
│   커밋 5개
│   [승인] [거절] 버튼
```

**검토 프로세스:**
1. 머지 요청 클릭하여 상세 내용 확인
2. 포함된 커밋 리스트 검토
3. 판단:
   - **승인**: 작업이 완료되고 품질이 충분함 → [승인] 버튼 클릭
   - **거절**: 추가 작업이 필요함 → [거절] 버튼 클릭 + 피드백 전달

#### 2.7. 머지 실행 (승인 시)
1. [승인] 버튼 클릭
2. 자동으로 머지 실행:
   - Feature-A의 모든 커밋이 Development 브랜치로 복사됨
   - Feature-A 브랜치 상태가 "merged"로 변경
   - Development 브랜치에 반영됨
3. 화이트보드에서 시각적으로 확인 가능
   - Feature-A 노드가 반투명해짐 (merged 상태)
   - Development 브랜치에 새 커밋들이 추가됨

---

### Phase 4: 상위 브랜치로 계층적 머지

#### 2.8. 부서 브랜치 → 회사 메인 브랜치
**상황:** Development 브랜치의 주간 작업을 회사 메인 브랜치로 반영

1. Development 브랜치 클릭하여 상세 페이지 이동
2. **"Merge to Parent"** 버튼 클릭 (타겟: Main)
3. CTO/임원 승인 대기
4. 승인되면 회사 메인 대시보드에 반영
   - 전체 팀원이 진행상황 확인 가능
   - 회사의 마일스톤 달성도 업데이트

## 3. 실제 사용 시나리오

### 시나리오 1: 개발자의 하루

**오전 9:00 - 브랜치 확인**
1. WorkHub 접속
2. 자신이 담당하는 "Login-Feature" 브랜치 상태 확인
3. 어제까지의 커밋 히스토리 검토

**오후 2:00 - 중간 커밋**
1. "로그인 UI 컴포넌트 완성"
2. 커밋 작성:
   - 제목: "로그인 폼 UI 구현 완료"
   - 설명: "이메일, 비밀번호 입력 필드 및 로그인 버튼 구현. 반응형 디자인 적용."
3. 커밋 → 팀원들이 진행상황 확인 가능

**오후 6:00 - 일일 커밋**
1. "로그인 유효성 검사 추가"
2. 커밋 작성 후 퇴근

**금요일 오후 - 주간 작업 완료**
1. 총 10개의 커밋 완료 확인
2. "Merge to Parent" 클릭 (타겟: Development)
3. 팀장 검토 요청
4. 팀장 승인 → Development 브랜치에 반영
5. 주간 보고서 자동 생성 (향후 기능)

---

### 시나리오 2: 마케팅 팀장의 업무

**월요일 오전 - 주간 계획**
1. WorkHub에서 Marketing 브랜치 확인
2. 하위 브랜치 확인:
   - Campaign-1 (SNS 광고 캠페인)
   - Campaign-2 (이메일 마케팅)
3. 각 팀원의 이번 주 목표 확인

**수요일 오후 - 검토 작업**
1. Pull Requests에서 "Campaign-1 → Marketing" 요청 확인
2. 팀원이 작성한 5개 커밋 검토:
   - "광고 소재 초안 작성"
   - "타겟 오디언스 분석 완료"
   - "예산안 작성"
   - "A/B 테스트 계획 수립"
   - "광고 런칭 준비 완료"
3. 내용 확인 후 [승인]
4. Campaign-1의 작업이 Marketing 브랜치에 반영됨

**금요일 오후 - 주간 보고**
1. Marketing 브랜치의 이번 주 전체 커밋 확인
2. "Merge to Parent" 클릭 (타겟: Main)
3. CEO 승인 요청
4. 승인되면 회사 전체 대시보드에 마케팅팀 성과 반영

---

### 시나리오 3: CEO의 모니터링

**매일 오전 - 전체 현황 파악**
1. WorkHub 메인 화면 접속
2. 브랜치 화이트보드에서 전체 구조 확인:
   ```
   Main (회사)
   ├── Development (개발팀)
   │   ├── Frontend (프론트엔드) - active
   │   └── Backend (백엔드) - active
   ├── Marketing (마케팅팀)
   │   └── Q1-Campaign - pending review
   └── Sales (영업팀)
       └── Partnership - active
   ```
3. 각 브랜치의 최근 커밋 수 확인
4. Pull Requests에서 승인 대기 중인 항목 확인

**주간 회의 전 - 머지 승인**
1. 각 부서장이 올린 Pull Request 검토
2. Development → Main 머지 승인
3. Marketing → Main 머지 승인
4. 회사 메인 브랜치에 이번 주 성과 반영
5. 주간 회의에서 누적된 커밋을 기반으로 보고

## 4. 주요 화면별 기능

### 4.1. WorkHub 메인 화면
**상단 헤더:**
- 연결된 프로젝트 정보
- [공시 데이터 업로드] 버튼 - 외부 프로젝트 공시 사항 불러오기
- [새 브랜치] 버튼 - 새로운 작업 브랜치 생성
- [커밋하기] 버튼 - 빠른 커밋

**Branch Overview:**
- 브랜치 트리 시각화 (화이트보드)
- 각 노드 클릭 → 브랜치 상세 페이지
- 색상으로 상태 표시:
  - 초록색 테두리: active (활성)
  - 노란색 테두리: pending (검토 대기)
  - 회색 반투명: merged (병합 완료)

**Pull Requests:**
- 모든 머지 요청 목록
- 상태별 필터링 가능
- 승인/거절 버튼 (권한 있는 사용자만)

**Current Branch:**
- 현재 선택된 브랜치 정보
- 타입, 멤버 수 표시

---

### 4.2. 브랜치 상세 페이지
**상단:**
- [← Back to Whiteboard] 버튼
- 브랜치 이름, 타입, 상태
- [Merge to Parent] 버튼

**커밋 입력창:**
- 제목 입력 (한 줄)
- 설명 입력 (여러 줄)
- [커밋하기] [취소] 버튼

**Commit History:**
- 시간 역순으로 커밋 리스트
- 각 커밋:
  - 날짜
  - 제목
  - 작성자
  - 상태 (draft/published/approved/merged)

## 5. 권한 및 역할

- ✅ 자신이 속한 브랜치 조회
- ✅ 커밋 작성

- ❌ 다른 팀원의 커밋 수정/삭제
- ❌ 머지 승인/거절

- ✅ 새 하위 브랜치 생성
- ✅ 팀원 추가/제거
- ❌ 상위 브랜치로의 머지 승인

### CEO/최고 관리자
- ✅ 모든 브랜치 조회
- ✅ 모든 머지 요청 승인/거절
- ✅ 회사 메인 브랜치 관리
- ✅ 최상위 브랜치 생성/삭제

## 6. 업무 팁

### 효과적인 커밋 작성법
**좋은 예:**
```
제목: 사용자 프로필 페이지 API 연동 완료
설명:
- GET /api/profile 엔드포인트 연동
- 사용자 정보 화면 표시 로직 구현
- 에러 처리 및 로딩 상태 추가
- 테스트 케이스 3건 작성
```

**나쁜 예:**
```
제목: 작업함
설명: 오늘 일 다 했음
```

### 브랜치 관리 팁
1. **세분화**: 큰 작업은 여러 하위 브랜치로 분할
2. **정기적 머지**: 주 1회 이상 상위 브랜치로 머지
3. **명확한 이름**: 브랜치 이름은 작업 내용을 명확히 표현
4. **완료 후 정리**: 머지된 브랜치는 아카이브

### 팀 협업 팁
1. **일일 커밋**: 하루 종료 시 반드시 커밋
2. **상세한 설명**: 다른 팀원도 이해할 수 있도록 작성
3. **빠른 검토**: Pull Request는 24시간 내 검토
4. **피드백 반영**: 거절 시 피드백을 참고하여 재작업

## 7. 자주 묻는 질문 (FAQ)

### Q1. 커밋을 잘못 작성했어요. 수정할 수 있나요?
A: 현재 버전에서는 커밋 수정 기능이 없습니다. 새로운 커밋으로 정정 내용을 작성해주세요.

### Q2. 브랜치를 삭제하고 싶어요.
A: 브랜치 삭제 기능은 관리자에게 요청해주세요. (향후 기능 추가 예정)

### Q3. 머지 요청이 거절되었어요. 어떻게 하나요?
A: 거절 사유를 확인하고, 브랜치에서 추가 작업을 완료한 후 다시 머지 요청하세요.

### Q4. 다른 팀의 브랜치를 볼 수 있나요?
A: 공개(public) 프로젝트만 볼 수 있습니다. 비공개 프로젝트는 권한이 있는 팀원만 접근 가능합니다.

### Q5. 커밋을 삭제할 수 있나요?
A: 보안 및 감사 목적으로 커밋 삭제는 불가능합니다. 잘못된 내용은 새 커밋으로 정정하세요.

## 부록: 용어 사전

| 용어 | 설명 |
|------|------|
| **브랜치 (Branch)** | 업무 단위를 나타내는 작업 공간 |
| **커밋 (Commit)** | 완료된 작업을 기록하는 행위 (공시) |
| **머지 (Merge)** | 하위 브랜치를 상위 브랜치로 통합 |
| **Pull Request (PR)** | 머지 요청 (검토 요청) |
| **승인 (Approve)** | 머지 요청을 허가하여 통합 실행 |
| **거절 (Reject)** | 머지 요청을 반려 |
| **메인 브랜치 (Main)** | 회사 전체의 최상위 브랜치 |
| **부모 브랜치 (Parent)** | 현재 브랜치의 상위 브랜치 |
| **자식 브랜치 (Child)** | 현재 브랜치의 하위 브랜치 |
| **공시 (Disclosure)** | 외부 프로젝트에서 공개된 정보를 가져오는 것 |
